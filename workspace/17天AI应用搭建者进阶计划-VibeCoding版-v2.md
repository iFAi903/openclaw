# 🎓 AI应用搭建者进阶计划（Vibe Coding × AI协作系统版）

> 基于原《17天AI应用搭建者进阶计划-VibeCoding版》升级
> 这次迭代不是简单加内容，而是补齐原计划里相对薄弱的“系统层”：
> **模型选择、Skill、Agent、MCP、Claude Code / OpenClaw、上下文工程、安全治理、产品化路径**
> 
> 适合对象：
> - 已有 Prompt / Vibe Coding 基础
> - 不满足于“会用 AI”，而想真正搭系统的人
> - 想从“搭一个 demo”进化到“构建长期可复用 AI 工作流”的人

---

# 一、这次迭代为什么必要

原 17 天计划的优点很明显：
- 已经抓住了 Vibe Coding 的灵魂
- 强调规划优先、上下文优先、Glue Coding
- 对 Prompt、Canvas、Context 传递已经有实操框架

但它也有几个天然盲区：

## 原计划的 5 个结构性缺口

### 1. 缺少“模型层”
原计划大量讲 Prompt 和 Context，但对：
- Opus / Sonnet / Haiku 怎么选
- 什么任务该用什么模型
- 为什么模型选择是第零步

讲得不够。

### 2. 缺少“Skill 层”
原计划强调 Prompt 模板和工作流，但没有把：
- 可复用方法论
- 个人风格
- SOP
- 长期复用资产

系统性上升为 **Skill** 概念。

### 3. 缺少“Agent × MCP”的清晰分工
原计划提到多 Agent、Context、工作流，但没有把：
- Agent 是“决策层”
- MCP 是“连接层”
- Workflow / Prompt Chain / Agent 各自适用边界

讲透。

### 4. 缺少“产品化载体”认知
原计划更像“方法训练营”，但现在你已经实际在用：
- OpenClaw
- 多 Agent 团队
- 长期记忆
- 消息通道

所以学习计划必须接上真实操作台，而不是停留在概念层。

### 5. 缺少“安全治理”作为独立能力
原计划讲了 Context，却还没有把：
- 权限边界
- 高自主 Agent 的红黄绿灯
- 什么时候必须 HITL（人在回路）

作为长期必修能力写进去。

# 三、阶段设计总览

## Phase 1｜Vibe 基础重建（Day 1-4）
目标：
不是重新学一遍入门，而是把原计划的基础压实成“系统设计底座”。

### Day 1｜Vibe Coding 哲学：规划不是前菜，是主菜
- 回顾原版 Day 1
- 新增重点：
  - 为什么“先结构后代码”不是效率低，而是长期最快
  - 区分：写代码 vs 设计信息结构
  - 用 Mermaid 画出一个 AI 应用的“目的—模块—数据流”三层图

**产出**：
- 1 张 AI 应用三层架构图
- 1 份“一句话目标 + 非目标 + 成功标准”定义卡

### Day 2｜上下文工程入门：Prompt 不够，考场才重要
- 回顾原版 Prompt 设计
- 新增重点：
  - Prompt Engineering 和 Context Engineering 的区别
  - 确定性上下文 vs 概率性上下文
  - 为什么“垃圾上下文”比“烂措辞”更致命

**实践**：
- 把同一个任务做 3 个版本：
  - 只给 Prompt
  - Prompt + 背景文档
  - Prompt + 背景文档 + 输出规范
- 对比质量变化

### Day 3｜Glue Coding：从“自己写”到“会组装”
- 保留原版核心
- 新增重点：
  - 哪些模块适合抄现成轮子
  - 哪些模块必须自己定义边界
  - 为什么“连接方式”比“底层细节”更重要

**实践**：
- 选一个开源项目
- 不改内核，只改：模型接口 / 输出格式 / 数据接入方式

### Day 4｜模型是第零步：Opus / Sonnet / Haiku 怎么选
这是原计划最该补的一天。

**学习目标**：
- 理解“模型选择是引擎选择”
- 建立任务复杂度 × 成本 × 速度 × 上下文规模的判断框架

**要点**：
- Haiku：轻量、快、便宜
- Sonnet：主力工作模型
- Opus：深度推理 / 大任务 / 复杂 review

**实践**：
- 列出自己 10 类常见任务
- 为每类任务指定默认模型
- 做一张“模型路由表”

**产出**：
- 个人模型路由策略 v1

## Day 5｜缓冲复盘 ①
复盘问题：

- 我是否已经开始按任务类型选模型？
- 我是否还在下意识自己写，而不是先找能接上的轮子？

**产出**：
- Phase 1 复盘卡
- 原有工作流的 3 个可替换点

## Phase 2｜Prompt × Context × Skill（Day 6-8）
目标：
把“会写提示词”，升级成“会沉淀方法论资产”。

### Day 6｜Prompt 进阶：System Prompt 作为 AI 的行为宪法
- 拆解优秀 System Prompt 的构成
- 学习角色、边界、输出格式、few-shot、反例约束
- 认识：强 Agent 的危险，不在于能力，而在于方向错

**实践**：
- 写一个“产品分析师”System Prompt
- 写一个“知识整理助手”System Prompt
- 写一个“严格 JSON 输出器”System Prompt

### Day 7｜Skill 思维：把高频方法论变成可复用资产
这一天是原计划最重要的新增。

**学习目标**：
- 理解 Prompt、模板、Skill 的区别
- 判断什么时候该把方法论从对话里抽出来
- 初步理解 `SKILL.md` 的价值

**实践**：
从你自己的工作里抽 2 个 Skill 候选：
- 比如：文章拆解法
- 比如：产品需求梳理法
- 比如：资料整理成 Obsidian 笔记的方法

为每个候选写：
- 使用场景
- 触发条件
- 输出格式
- 禁止行为

**产出**：
- 2 个个人 Skill 草案

### Day 8｜Skill 落地：从方法论到笔记、模板、工作流
**目标**：
让 Skill 不停留在概念层，而是能接进你自己的知识库。

**实践**：
- 选择一个 Skill 草案
- 把它拆成：
  - 规则
  - 模板
  - 示例
  - 检查清单
- 存入 Obsidian
- 为它建立双链结构

**产出**：
- 第一个可复用的 Obsidian 方法论 Skill 笔记

---

## Phase 3｜Agent × Workflow × MCP（Day 9-12）
目标：
建立对“什么时候需要 Agent，什么时候不需要”的清晰判断。

### Day 9｜Workflow、Prompt Chain、Agent 的边界
**核心问题**：
不是“Agent 很酷所以都用 Agent”，而是：
- 单次 Prompt 够不够
- Prompt Chain 是否更稳
- Workflow 是否更可控
- Agent 是否真的必要

**实践**：
用同一任务分别设计：
- 单 Prompt 版
- Prompt Chain 版
- Workflow 版
- Agent 版

比较：
- 成本
- 稳定性
- 灵活性
- 风险

### Day 10｜Agent 的本质：规划、记忆、工具调用、反思
**学习目标**：
- 理解 Agent 不只是“会聊天的 AI”
- 知道 Agent 的四个核心组件
- 理解为什么 Agent 越强越需要边界

**实践**：
用一个真实任务拆解 Agent 回路：
Observe → Plan → Act → Reflect

### Day 11｜MCP：AI 时代的标准连接器
这是另一块关键补丁。

**学习目标**：
- 理解 MCP 不是“更好的 API”，而是“AI 原生接口层”
- 理解 Tools / Resources / Prompts 三原语
- 理解 MCP 的价值在标准化，而不只是连接

**实践**：
选择 3 个你最需要接入的外部世界：
- 文档
- 数据库 / 表格
- 搜索 / 浏览器

分别写出：
- 接入目的
- 预期输入输出
- 风险等级

### Day 12｜Agent × MCP 实战设计：什么时候该接，什么时候别接
**学习目标**：
- 不是一切都接外部工具
- 理解“外部能力越强，治理要求越高”

**实践**：
设计一个小型方案：
“资料收集 → 整理 → 输出总结”
分别画出：
- 无 MCP 版本
- 有 MCP 版本
- 两者的优缺点

**产出**：
- 一张 Agent × MCP 适用边界图

## Day 13｜缓冲复盘 ②

- 我什么时候真的需要 Agent？
- 我的哪些流程其实用 Workflow 更合适？
- 如果我接外部世界，哪些是绿灯，哪些是黄灯，哪些是红灯？

## Phase 4｜Claude Code × 模型路由 × Coding Agent（Day 14-16）
目标：
把前面的方法论真正带进“能做事的工作台”。

### Day 14｜Claude Code 的本质：不是聊天，而是到工地干活
**学习目标**：
- 理解 Claude Code 和普通聊天界面的本质差异
- 理解 CLAUDE.md / 项目级规范 / 子 Agent / Agent Teams

**实践**：
- 找一个小项目
- 用自然语言让 Claude Code 读结构、提方案、做改动

### Day 15｜Vibe Coding 进阶：描述需求，不是微操代码
**核心点**：
Vibe Coding 不是“不会编程也能碰运气”，而是：
- 你用更高层语言定义目标
- AI 负责实现
- 你负责审查与校准

**实践**：
- 让 AI 做一个小功能
- 你只提供：目标、非目标、验收标准、可改范围
- 记录效果

### Day 16｜模型路由实战：规划用谁，执行用谁，审查用谁
**学习目标**：
把 Day 4 的模型理解落到真实工作流。

**实践**：
为一个真实开发任务设计三段式：
- 规划：Opus / Sonnet？
- 执行：Sonnet / Haiku？
- 审查：Opus / Sonnet？

**产出**：
- 个人 Coding Agent 工作流 v1

### Day 18｜长期记忆设计：什么该记，什么不该记
**学习目标**：

# 四、这版计划相对原版新增的关键模块

## 新增模块 A｜模型路由
原版弱，这版补强。
原因：
- 模型选择决定任务上限
- 成本和速度在真实工作流里是刚需

## 新增模块 B｜Skill 方法论资产化
原版偏 Prompt，这版把 Skill 拉出来单列。
原因：
- 你需要的不是一次次“重新说”，而是把方法沉淀成可复用资产

## 新增模块 C｜Agent / Workflow / MCP 的边界判断
原因：
- 很多人不是不会用 AI，而是乱上 Agent
- 乱接外部世界，比不会接更危险

## 新增模块 D｜Claude Code / OpenClaw 的产品化理解
原因：
- 你现在已经不只是学习者，而是在搭真实系统
- 所以学习计划必须接地

## 新增模块 E｜安全治理层
原因：
- 自主系统没有刹车，就不是系统，是事故预备队
