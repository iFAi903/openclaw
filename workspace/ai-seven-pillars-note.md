> 整理目标：压缩为可复习、可引用、可执行的 Obsidian 笔记
> 整理时间：2026-03-11

---

## 一句话总览

AI 协作不是“找一个最强工具”，而是搭一个**分层协作系统**：

- **大模型**是引擎
- **Prompt** 是语言
- **Skill** 是方法论封装
- **Agent** 是决策执行体
- **MCP** 是外部连接器
- **Claude Code** 是按需启动的编码操作台
- **OpenClaw** 是 24/7 在线的数字分身
- **上下文工程** 是统合这一切的操作系统

---

## 核心结论

### 1. 七大组件不是替代关系，而是分层关系
它们更像一个 **stack（栈）**，不是一个 **menu（菜单）**。

很多人的误区，是把：
- Prompt
- Agent
- MCP
- Skill

看成同级竞品，仿佛只能选一个。

其实不是。
它们分别解决不同层的问题：

- Prompt：怎么说
- Skill：怎么复用
- Agent：怎么自己决定步骤
- MCP：怎么连接外部世界
- 上下文工程：怎么让所有信息在正确时刻出现

### 2. 提示词工程没有死，只是被纳入更大的框架
2026 年的升级不是“Prompt 过时”，而是：

> Prompt Engineering → 被吸收进 Context Engineering

也就是说，问题不再只是“怎么措辞”，而是：
- 给 AI 看什么资料
- 以什么顺序给
- 给多少
- 哪些是确定性规则
- 哪些是动态检索结果

### 3. 模型选择是被严重低估的“第零步”
不是所有任务都该上最强模型。

更正确的判断方式：
- **Haiku**：轻任务、速度优先、成本敏感
- **Sonnet**：大多数日常工作主力
- **Opus**：深度推理、架构级任务、复杂审查

关键洞见：

> “够用”常常比“最强”更重要。

### 4. 安全是整套系统中唯一真正的负反馈机制
系统越自动化，风险越高。

尤其当 Agent 接上外部工具后：
- Prompt 注入
- 工具投毒
- 越权操作
- 数据泄露

都不是边缘问题，而是系统级问题。

所以最好的设计通常不是“最大自主”，而是：

> 给最强的大脑，但只给最小的手脚。

## 七大基柱速记

## 0. 大模型（Foundation Model）
### 定义
底层推理引擎，决定能力上限。

### 它解决什么
- 推理深度
- 编码能力
- 上下文容量
- 成本/速度平衡

### 心智模型
像汽车发动机：
- Opus = V12
- Sonnet = V6
- Haiku = 轻快四缸

### 关键提醒
选错模型，不是“差一点”，可能是“根本做不到”。

---

## 1. Prompt
### 定义
你和 AI 沟通的基础语言。

### 它解决什么
- 传达意图
- 限制边界
- 说明输出格式
- 提供任务上下文

### 关键提醒
所有更高层结构，最终都要翻译成 Prompt 送给模型。

---

## 2. Skill
### 定义
把高频可复用的方法论，封装成 AI 可调用的能力模块。

### 它解决什么
- 反复输入同类指令的问题
- 稳定输出风格和流程
- 团队共享方法论

### Skill 本质上包含
- `SKILL.md`
- 模板
- 规则
- 脚本
- 支撑资源

### 关键提醒
如果你连续很多次都在给 AI 讲同一套做法，基本就该做 Skill 了。

---

## 3. Agent
### 定义
能够自主规划、分步执行、调用工具、检查结果的 AI 系统。

### 它解决什么
- 多步骤任务
- 步骤数量不固定
- 需要自己判断下一步怎么走

### Agent 与普通聊天的区别
普通聊天：
- 你说一步，它做一步

Agent：
- 你给目标，它自己拆路径

### 关键提醒
Agent 不适合解决所有问题。
如果任务步骤本来就确定，Workflow 往往更稳、更便宜。

---

## 4. MCP
### 定义
AI 连接外部工具和数据源的标准协议。

### 它解决什么
- 连接 GitHub、数据库、文档、搜索、服务 API
- 避免每个模型都对每个工具做单独适配

### 最好理解的比喻
MCP 之于 AI，像 USB-C 之于设备。

### 关键提醒
MCP 的核心价值不只是“能连”，更是“标准化红利”。

---

## 5. Claude Code
### 定义
Anthropic 的编码型 Agent 产品化形态。

### 它解决什么
- 读代码库
- 改文件
- 跑命令
- 测试
- Git 协作
- 项目级自动化

### 本质
它不是“更会聊天的 Claude”，而是：

> AI 直接坐到你的项目现场开始干活。

### 关键提醒
Claude Code 证明了一件事：
七大基柱不是概念，而是已经被成熟产品验证的工程现实。

---

## 6. OpenClaw
### 定义
一个本地运行、24/7 在线、多通道接入、带持久记忆的个人 AI 分身。

### 它解决什么
- 永久在线
- 跨消息平台接入
- 长期记忆
- 定时任务与主动执行
- 多 Agent 路由与长期陪伴

### 本质差异
Claude Code：按需唤醒，用完即停
OpenClaw：持续在线，一直存在

### 关键提醒
OpenClaw 的价值和风险是一起放大的：
- 自主性越强，攻击面越大
- 权限越广，治理越难

所以它更像“高阶系统”，不是所有人都该直接上来就全开权限。

---

## 7. 上下文工程（Context Engineering）
### 定义
围绕“让 AI 在正确的信息环境中思考”的整体设计能力。

### 它解决什么
- 给什么上下文
- 什么时候给
- 给多少
- 哪些信息优先级更高
- 如何管理记忆、检索、规则、工具返回结果

### 核心洞见
Prompt 是 Context 的一部分，不是全部。

### 最实用的理解方式
提示词工程是“怎么说话”，
上下文工程是“怎么布置整个考场”。

---

## 一个非常好用的总类比：硅基生命体

如果把整套 AI 系统想象成一个“硅基生命体”：

- **大模型** = 大脑
- **Agent** = 身体
- **上下文工程** = 操作系统 / 记忆系统
- **Skill** = 内功心法
- **MCP** = 外部感官
- **Claude Code** = 精英雇佣兵
- **OpenClaw** = 贴身管家
- **你** = 灵魂

这套类比最有价值的地方在于：

> AI 的真正战斗力，不只取决于模型，而取决于“你怎么组织这具硅基生命体”。

## 最实用的决策框架

## 1. 从简单到复杂的升级路径
优先顺序建议：

1. 单次 Prompt
2. Prompt Chain / Workflow
3. Skill
4. Agent
5. Agent + MCP
6. Claude Code / OpenClaw 等产品化载体

一个非常重要的原则：

> 先用最简单的方法解决，真的不够，再升级复杂度。

---

## 2. 四象限选择法
用两个维度判断：
- 任务复杂度
- 重复频率

### 低复杂 + 一次性
用 **Prompt**

### 低复杂 + 高频
用 **Skill**

### 高复杂 + 一次性
用 **Agent + MCP（按需）**

### 高复杂 + 高频
用 **Agent + Skill + MCP 的全栈组合**

---

## 3. 安全红绿灯
### 绿灯
可自主执行：
- 搜索
- 读取
- 汇总
- 内部草稿
- 可逆操作

### 黄灯
执行前最好确认：
- 外部发信
- 发布内容
- 修改记录
- 消耗资源
- 影响他人的写操作

### 红灯
必须人工审批：
- 转账
- 删除关键数据
- 改生产配置
- 处理高敏信息
- 不可逆系统变更

## Claude Code、Cowork、OpenClaw 的关系

原文后半还扩展到一个很重要的产品层对比：

### Claude Code
- 面向程序员
- 按需启动
- 强于编码、项目操作、系统级执行
- 特点：精控、受控、专业

### Cowork
- 面向知识工作者
- 强于办公流、文档流、跨应用处理
- 更像“知识工作台”

### OpenClaw
- 面向想要 24/7 在线分身的人
- 强于长期记忆、多通道、持续在线、自主执行
- 自由度最高，也最需要安全边界

一句话概括：

> Claude Code 是工作台，Cowork 是办公台，OpenClaw 是前哨站。

## 常见误区

### 误区 1：想找一个万能 Prompt
现实是：
很多问题不是措辞问题，而是上下文不够、结构不对。

### 误区 2：什么都想上 Agent
很多任务根本不需要 Agent。
步骤确定的任务，Workflow 往往更稳。

### 误区 3：把 Skill 当成超长 Prompt
Skill 不是“长一点的提示词”，而是：
- 封装
- 结构化
- 渐进加载
- 可复用

### 误区 4：安全以后再说
对于高权限 Agent，这种想法几乎等于埋雷。

### 误区 5：永远选最强模型
很多场景下，最贵并不最优。
模型路由才是长期正确策略。

### 误区 6：把 OpenClaw 和 Claude Code 当同类竞品
它们的存在方式不同：
- 一个是按需工具
- 一个是常驻实体

## 我认为这篇资料最值得记住的 10 句话

1. 七大组件构成的是一个栈，不是一个菜单。
2. Prompt 没有过时，它只是被纳入了更大的上下文工程。
3. 模型选择是被低估的第零步。
4. 够用比最强更重要。
5. Agent 越强，System Prompt 越重要。
6. Skill 的本质是“写一次，复用无限次”。
7. MCP 的价值不只在连接，更在标准化。
8. 安全不是附属功能，而是系统制动器。
9. Claude Code 是七层协作栈的产品化证明。
10. 你给 AI 的上下文质量，决定它在你手里的战斗力。

## 适合放进个人方法论里的行动建议

### 如果你是内容创作者
先做：
- 风格 Skill
- 选题 Prompt 模板
- 资料检索流程

### 如果你是产品经理 / 策略工作者
先做：
- 分析框架 Skill
- 竞品调研 Agent 流程
- 文档知识库连接

### 如果你是技术/产品混合型创业者
优先学：
- 模型路由
- Claude Code
- MCP
- 上下文工程

### 如果你想做长期在线的数字分身
重点关注：
- OpenClaw 架构
- 权限边界
- 记忆治理
- 通道路由
- 安全红绿灯

## 最终总结

这份资料最有价值的，不是罗列一堆新名词。

它真正讲清楚的是：

> AI 的能力，不只取决于模型本身，而取决于你是否能把模型、上下文、方法论、连接器、执行体，组织成一个稳定协作的系统。

所以，未来真正拉开差距的，不只是“谁会用 AI”，而是：

- 谁更会选模型
- 谁更会设计上下文
- 谁更会封装方法论
- 谁更会划安全边界
- 谁更会把 AI 变成一套长期可复用的操作系统
