# Website Ops Safe

把“网站运营”视为一条闭环链路，而不是单次动作。

核心原则：
1. 先确认数据或内容是否真的生成
2. 再确认代码或构建源是否真的更新
3. 再确认部署是否真的成功
4. 最后确认主域名是否真的对用户可见
5. 不把本地成功当成线上成功
6. 不把 deploy 返回成功当成用户可见成功
7. 不默认对外发布；任何外部平台发文/发帖/群发都必须先征得用户同意
8. 不安装未经审计的第三方 skill、脚本或发布器

## 标准工作流

### Phase 1：确认源头
检查：
- 内容/数据是否已经生成
- 生成时间是否为本次目标日期/版本
- 页面或应用实际读取的源文件是哪一个
- 是否存在“更新了 daily_data.json，但页面读的是 news.ts”这类错位

### Phase 2：确认构建源
检查：
- 页面入口文件读的是哪份数据
- 构建脚本是否会把最新数据打进产物
- 本地 build 是否通过
- 是否有多 lockfile / workspace root / 路径空格等潜在风险

### Phase 3：确认部署链路
检查：
- 部署目标项目是否正确
- 环境变量是否就位
- 生产部署是否返回成功
- 主域名 alias 是否指向最新部署

### Phase 4：线上验收
至少做两件事：
- 打开主域名检查日期/标题/关键内容
- 抽样给用户 2-6 个可点击验证链接

只有这一步完成，才叫“上线完成”。

## 输出要求

网站更新类任务完成时，默认给出：
1. 主站链接
2. 当前线上日期/版本
3. 2-6 条可点击内容链接
4. 结论：已上线 / 本地已更新但线上未生效 / 部分成功
5. 如有异常，指出卡在 数据 / 构建 / 部署 / alias / 缓存 哪一层

## 安全边界

### 默认允许
- 读取项目代码与构建配置
- 运行本地 build
- 检查部署状态
- 验证主域名页面
- 整理检查清单与运营工作流

### 必须谨慎
- 修改生产部署配置
- 修改 cron / heartbeat / 自动化任务
- 接入新 API / 新 token / 新 webhook

### 默认禁止（除非用户明确授权）
- 自动外部发文、发帖、群发、回帖
- 安装未经审计的第三方发布技能或自动化脚本
- 扩大到平台账号接管级权限
- 读取或复制无关凭据

## 第三方技能/方案审计规则

遇到外部文章、GitHub、ClawHub、博客教程时：
1. 学习方法论可以
2. 直接照搬自动化链路不可以
3. 涉及外部发布/登录/凭据/抓 cookie 的内容一律提高风险等级
4. 先做 source check + code review + permission scope review，再决定是否安装

## 推荐配套产物

执行这类任务时，优先产出：
- `last_run_status.json` 或等价状态文件
- 持久日志（不要只写 /tmp）
- 发布前检查清单
- 部署后验收记录

## 快速验收清单

- 数据是今天的吗？
- 页面入口读的是这份数据吗？
- 本地 build 通过了吗？
- 生产部署成功了吗？
- 主域名已切到新版本了吗？
- 我已经给用户可点击验证链接了吗？

## 何时触发这个 skill

当用户提到以下任一场景时使用：
- 网站更新、上线、部署、没生效
- 博客/日报/新闻站/SEO 页面发布
- 定时任务生成后要检查线上是否更新
- 本地和线上版本不一致
- 想给 Agent 配置“安全版网站运营能力”
