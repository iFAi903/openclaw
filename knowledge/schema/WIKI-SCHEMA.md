# WIKI-SCHEMA

> 本文件定义《小羽毛 Wiki OS》的核心结构与维护规则。

---

## 1. 系统目标

小羽毛 Wiki OS 的目标不是存档，而是持续编译知识。

系统要做到：

- 把原始资料转化为结构化页面
- 把高价值问答沉淀为长期资产
- 让知识页面之间形成可追踪、可更新、可交叉引用的网络
- 让小羽毛天团围绕同一套知识底座协作

---

## 2. 三层结构

### Raw
只读原始资料层：
- 不直接承载结论
- 不做人工改写
- 仅作为事实来源

### Wiki
知识编译层：
- 由 LLM 与天团维护
- 负责总结、抽象、交叉引用、更新
- 面向查询与长期积累

### Schema
规则治理层：
- 规定页面类型、流程、命名和维护方式
- 保证跨 session 一致性

---

## 3. 页面类型

允许的核心页面类型：

- `entity`：人物 / 品牌 / 公司 / 产品 / 项目
- `concept`：概念 / 主题 / 抽象结构
- `method`：方法 / SOP / 工作流 / 框架
- `decision`：关键决策与判断
- `synthesis`：综合分析 / 比较 / 阶段性总结
- `timeline`：演化过程 / 时间线

---

## 4. 核心流程

### Ingest
新增 raw source 后，至少完成以下动作：
1. 判断主题与归属目录
2. 提炼关键内容
3. 更新 1 个以上相关 wiki 页面
4. 更新 `wiki/index.md`
5. 在 `wiki/log.md` 追加记录

### Query
回答问题时，优先流程：
1. 从 `index.md` 找相关页面
2. 读页面
3. 必要时回溯 raw
4. 输出答案
5. 判断是否值得回写为长期页面

### Lint
定期检查：
- 重复页面
- 孤儿页
- 断链
- 过时结论
- 缺失页面
- 索引未更新

---

## 5. 更新规则

### 必须更新 index 的情况
- 新建页面
- 页面重命名
- 页面合并
- 页面归档

### 必须更新 log 的情况
- ingest
- query 后回写
- lint
- 大规模重构

### 必须做回写判断的情况
- 回答形成可复用框架
- 输出形成阶段性结论
- 多资料综合后得到高价值认知

---

## 6. 命名规范

- 文件名优先使用清晰主题名
- 一个核心概念原则上只保留一个主页面
- 近义重复页面要么合并，要么建立主从关系
- 中文页面名优先，必要时再加英文别名策略

---

## 7. 角色分工

### CEO
- 决定结构
- 审核高价值沉淀
- 维护主轴与优先级

### CRO
- 维护叙事、品牌、表达相关页面

### CTO
- 维护系统、工具、自动化与 schema 页面

### PLAN
- 维护决策、时间线、路线图相关页面

### REVIEW
- 负责 lint、查重、查矛盾、提治理建议

---

## 8. 沉淀标准

优先沉淀：
- 高复用价值内容
- 长期主题相关内容
- 跨资料整合后的判断
- 关键项目结论
- 方法论与框架

不优先沉淀：
- 一次性确认
- 低信息密度对话
- 纯执行型回执
- 临时无复用价值的沟通

---

## 9. 当前阶段规则

当前处于 Bootstrap 阶段，规则如下：

- 先保证结构清晰，再追求规模扩张
- 先人工维护，再决定自动化
- 每新增高价值页面，必须补全 index
- 每完成一轮重要更新，必须写 log
