# 🧠 小羽毛记忆治理准则 v1

> 建立时间：2026-03-26  
> 目的：将“记忆能用”升级为“记忆可治理、可验证、可协作”。  
> 适用对象：小羽毛 CEO 及整个 AI 天团。

---

## 一、这份准则要解决什么问题

记忆系统的核心，不是“能存能查”，而是以下五件事同时成立：

1. **准确**：存进去的是对的，不是误判或噪音
2. **可召回**：需要时能找到，而不是埋掉
3. **低噪音**：找的时候不要把无关内容抖出来
4. **有边界**：谁能读、谁能写、谁能更新要清楚
5. **可治理**：记忆有生命周期，不是一旦写入就永久漂浮

一句话：

> **好的记忆，不是记得多，而是记得准、记得分层、记得在正确的时候浮现。**

---

## 二、记忆治理的六条核心原则

### 原则 1：主记忆正常，不代表团队记忆正常
任何“记忆系统正常”的判断，都必须明确层级：
- 主会话记忆正常
- 团队分身记忆正常
- 跨会话协作记忆正常

禁止再用模糊表述把三者混为一谈。

### 原则 2：能检索，不代表检索得准
Recall 成功只说明系统可用；
Recall 排名正确、噪音可控，才说明系统质量合格。

以后必须分开评估：
- **链路可用性**
- **排名质量**

### 原则 3：Archive 不是 Delete
归档的语义是：
- 从默认自动召回中降级/退场
- 但不等于彻底删除
- 精准手动查询仍可能命中

所以：
- 想“暂时退场” → archive
- 想“彻底消失” → forget/delete

### 原则 4：测试记忆必须短命
所有自检、实验、哨兵、验证性写入，都必须遵守：
1. 唯一标识
2. 立即回读
3. 测试完成后删除或归档
4. 不允许让测试记忆长期污染正式记忆库

### 原则 5：配置修复不能靠猜结构
任何涉及配置层（OpenClaw config / plugin config / tools allowlist）的修复，都必须：
1. 先确认 schema 支持
2. 一次只改一个层级
3. 改完立即重启验证
4. 不允许凭“看起来像能写进去”直接下注

### 原则 6：记忆管理是“边界 + 分层 + 生命周期 + 召回质量”
以后不再把记忆系统理解为单纯的向量库。
它本质上是一个：
- 有权限边界的系统
- 有分层结构的系统
- 有生命周期的系统
- 有排序质量要求的系统

## 三、记忆的分层模型

### A. 核心身份层
存放：
- 小羽毛的身份、人格、角色边界
- CEO/分身的职责定义
- 长期不轻易改变的工作哲学

特点：
- 稳定
- 高权重
- 低频更新

### B. 用户偏好层
存放：
- Leo 的明确偏好
- 沟通风格要求
- 审美、执行、工具选择偏好
- 已确认的长期规则

特点：
- 高频调用
- 必须低噪音
- 必须高命中

### C. 项目上下文层
存放：
- 当前项目状态
- 关键决策
- 里程碑变化
- 项目相关规则

特点：
- 中高频更新
- 生命周期强
- 过期后要归档

### D. 临时事实层
存放：
- 一次性事实
- 临时执行状态
- 某次检查结果

特点：
- 短生命周期
- 可快速过期
- 不应长期占据高优先级召回位

### E. 反思与经验层
存放：
- 错误、教训、原则
- 执行模式总结
- 排障经验

特点：
- 少而精
- 要可复用
- 必须经过提炼后再存

## 四、写入规则

### 什么时候该写
以下内容应优先写入长期记忆：
- 用户明确表达的稳定偏好
- 项目级关键决策
- 错误与教训
- 会影响后续执行判断的原则
- 可复用的工作方法

### 什么时候不该写
以下内容默认不进长期记忆：
- 单次闲聊噪音
- 未确认的猜测
- 临时情绪波动
- 测试过程中的中间垃圾数据
- 大段未经提炼的对话流水

### 写入格式要求
每条记忆尽量满足：
- 短
- 原子化
- 带关键词
- 语义清楚
- 单条只表达一件核心事

避免：
- 一条里塞多个无关事实
- 空泛形容词堆砌
- 没有检索锚点的抽象大话

## 五、召回规则

### 召回时先判断 query 类型
#### 1. 明确 query
例如：
- Leo 偏好 稳定 保守 默认 配置
- 新闻早报 来源分散

策略：
- 直接 recall
- 预期高精度

#### 2. 抽象 query
例如：
- 小羽毛 CEO 天团 分身 总控
- 我的组织方式是什么

策略：
- 先拆 query
- 必要时多轮 recall
- 不因抽象 query 直接断言系统失效

### 召回结果评估标准
每次 recall 至少从三点看：
1. 第一名是否命中真正目标
2. Top 3 是否有明显噪音
3. 是否有应当出现却没出现的关键记忆

## 六、治理动作规则

### update
适用：
- 原事实修正
- 规则完善
- 偏好明确化

要求：
- 更新后立即回读验证
- 防止文本更新但检索锚点消失

### promote
适用：
- 已验证为高价值长期规则
- 用户明确确认的重要偏好
- 稳定有效的关键决策

要求：
- promote 后应能在 explain/rank 中看到状态提升

### archive
适用：
- 过期项目记忆
- 历史阶段性记录
- 不应继续默认浮现的旧上下文

要求：
- 明确承认：archive 不是 delete
- 如果希望彻底不见，要再走 forget/delete

### forget/delete
适用：
- 错误测试数据
- 不应保留的污染记忆
- 用户明确要求删除的条目

要求：
- 删除后再次 recall 验证

---

## 七、团队记忆权限原则

### 现阶段事实
- CEO 主会话具有完整记忆操作能力
- 团队子会话/分身不一定自动继承完整 memory CRUD
- 团队级自检必须单独验证，不能默认继承

### 权限设计原则
以后团队记忆权限要明确区分：
- 谁可读
- 谁可写
- 谁可更新
- 谁可归档
- 谁只能通过 CEO 间接写入

禁止再默认认为“分身天然有脑”。

---

## 八、体检机制

### 每周至少做一次四类体检
#### 1. 基础链路体检
- memory_stats
- memory_store
- memory_recall
- memory_forget

#### 2. 排名质量体检
- 明确 query 测试
- 抽象 query 测试
- 噪音评估

#### 3. 治理链路体检
- memory_update
- memory_promote
- memory_archive
- explain/debug

#### 4. 团队权限体检
- 主会话 vs 子会话
- 是否可写
- 是否串线
- 是否越权
