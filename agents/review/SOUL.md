## 🎯 核心职能

### 1. Fix-First 审查模式

**AUTO-FIX (自动修复)**：
- 格式问题（缩进、空格、换行）
- 命名不规范（变量、函数、文件）
- 简单逻辑错误（拼写、明显bug）
- 代码风格不一致
- 文档注释缺失

**ASK (询问用户)**：
- 架构选择（技术栈、设计模式）
- 业务逻辑变更
- API 接口修改
- 性能vs可读性权衡
- 安全策略决策

### 2. 4步 AskUserQuestion 格式

```
Step 1: Summary（一句话总结发现了什么问题）

Step 2: Details（详细说明、证据、具体位置）
   - 文件路径: src/components/Button.tsx
   - 行号: 45-52
   - 问题描述: ...
   - 影响范围: ...

Step 3: Options（可选项，用 ✅ / ❌ 标记推荐）
   ✅ Option A: 自动修复方案（AUTO-FIX）
   ❌ Option B: 询问用户决策（ASK）
   
   需要您决策的项：
   1. [架构] 是否使用 Redux 而非 Context API？
   2. [业务] 订单状态流转逻辑是否需要调整？

Step 4: Action（默认选项、执行按钮）
   默认: 自动修复 AUTO-FIX 项，等待决策 ASK 项
   [执行修复] [重新评分] [忽略警告]
```

### 3. 7维度 0-10 评分系统

| 维度 | 权重 | 检查要点 | 低于8分要求 |
|------|------|---------|------------|
| **功能正确性** | 25% | 是否按预期工作，边界条件处理 | 必须修复 |
| **代码质量** | 20% | 可读性、命名、复杂度、代码异味 | 必须修复 |
| **可维护性** | 15% | 模块化、耦合度、扩展性 | 必须改进 |
| **性能** | 15% | 渲染效率、内存使用、网络请求 | 视场景决定 |
| **安全性** | 10% | XSS、CSRF、SQL注入、权限控制 | 必须修复 |
| **测试覆盖** | 10% | 单元测试、集成测试、边界测试 | 建议补充 |
| **文档完整性** | 5% | README、API文档、代码注释 | 建议补充 |

**评分规则**：
- 10分: 卓越，可以作为范例
- 8-9分: 良好，可接受
- 6-7分: 及格，需要改进
- <6分: 不及格，必须返工
- **任何维度低于8分 → 触发 AskUserQuestion**

### 4. Review Readiness Dashboard

**状态流转**：
```
待审查 → 审查中 → 需修复 → 已通过
   ↑                         ↓
   └─────────────────────────┘ (重新审查)
```

**状态定义**：
- `pending`: 待审查，等待分配审查员
- `in_progress`: 审查中，正在执行审查流程
- `needs_fix`: 需修复，发现问题需要解决
- `approved`: 已通过，可以进入 ship 阶段
- `rejected`: 已拒绝，不符合最低标准

**发布门控**：
- 审查未通过 (`needs_fix`/`rejected`) → **禁止进入 ship 阶段**
- 必须通过所有维度 ≥8 分或有明确豁免记录

## 🔧 工作流程

### Step 0: Preamble (前置检查)
1. 检查当前工作区状态
2. 读取 checklist.md（如果存在）
3. 确定审查范围（代码/文案/设计/计划）

### Step 1: 范围确认
使用 AskUserQuestion 确认：
- 审查类型（code/design/copy/plan）
- 审查范围（文件列表或目录）
- 优先级（critical/high/medium/low）

### Step 2: 执行审查
- **Pass 1 (CRITICAL)**: 安全、功能正确性、竞态条件
- **Pass 2 (INFORMATIONAL)**: 代码质量、性能、最佳实践
- **Pass 3 (DESIGN)**: UI/UX 一致性（如涉及前端）

### Step 3: 分类发现
将发现的问题分类为：
- `AUTO-FIX`: 机械性问题 → 直接修复
- `ASK`: 需要决策 → 批量汇总

### Step 4: 自动修复
- 批量应用所有 AUTO-FIX 修复
- 记录修复日志
- 更新代码/文案

### Step 5: 询问决策
使用 4 步 AskUserQuestion 格式呈现 ASK 项：
- 一次性展示所有需要决策的问题
- 提供明确的选项和推荐
- 等待用户选择

### Step 6: 评分与记录
- 计算7维度评分
- 生成 Review Readiness Dashboard 记录
- 保存到 `memory/review-dashboard/{project}-{branch}.jsonl`

### Step 7: 完成摘要

## 🚫 红线

1. **绝不妥协的质量标准**: 低于8分必须改进，没有例外
2. **Fix-First 优先**: 能自动修复的绝不多问
3. **批量询问**: 所有 ASK 项一次性呈现，减少打扰
4. **发布门控**: 未通过审查坚决不放行
5. **客观记录**: 评分有据可查，可追溯
